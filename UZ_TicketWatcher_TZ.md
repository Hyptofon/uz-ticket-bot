## 0. TL;DR

- Це **особистий** Telegram-бот (1 основний користувач, максимум кілька довірених людей) — НЕ SaaS на тисячі користувачів. Не треба мікросервісів, Kubernetes чи чергів повідомлень.
- Бот моніторить появу квитків на вказаний маршрут/поїзд/дату на `booking.uz.gov.ua` і сповіщає в Telegram.
- **Найважливіша вимога проєкту:** коли квиток знайдено, бот **не надсилає одне повідомлення і не зупиняється**. Він **бомбардує користувача повідомленнями** за розкладом ескалації, поки квиток не зникне або користувач сам не натисне "досить". Це навмисна поведінка — мета: не пропустити дефіцитний квиток, навіть якщо людина спить або телефон лежить в іншій кімнаті. Класична дедуплікація сповіщень **не використовується**. Повна специфікація — розділ 9.
- Бот **не купує** квитки автоматично — це робить сам користувач вручну одразу після сповіщення. В офіційному застосунку УЗ вже є авто-викуп; короткий порівняльний контекст — розділ 3.
- Джерело даних (`booking.uz.gov.ua`) — неофіційне, без документації, захищене від ботів (Cloudflare, reCAPTCHA). Розділ 7 описує, що про нього відомо із зовнішніх джерел, і що саме треба перевірити емпірично перед реалізацією.

## 1. Мета та контекст проєкту

**Проблема.** На популярних напрямках УЗ квитки розкуповують за секунди-хвилини після появи (повернення, додаткові вагони, відкриття продажу). Ручне оновлення сторінки вручну — неефективне і виснажливе.

**Рішення.** Персональний Telegram-бот, який:
1. Дозволяє через діалог у Telegram налаштувати відстеження конкретного маршруту (звідки → куди, дата, опційно номер поїзда, тип вагона).
2. У фоні регулярно перевіряє наявність місць.
3. При появі квитка — **негайно й потім багаторазово** сповіщає користувача, поки той не відреагує.

**Хто користувач.** Одна людина (автор бота), яка часто користується потягами і хоче отримати перевагу в швидкості реакції над іншими пасажирами, що чекають на ту саму дефіцитну поїздку.

## 2. Область застосування

### Входить в MVP
- Створення/перегляд/пауза/видалення моніторингів через Telegram-бот.
- Фоновий полінг наявності місць.
- Ескалаційна система сповіщень (розділ 9).
- Фільтрація за типом вагона та (за наявності в API) розташуванням місця.
- Стійкість до тимчасових збоїв API УЗ (retry, backoff).
- Базове логування.

### Явно НЕ входить в MVP (Phase 2 і далі, розділ 19)
- Автоматичний викуп/оплата квитків — бот тільки сповіщає, купівля залишається за користувачем вручну через сайт/застосунок УЗ.
- Веб-дашборд, мультиканальні сповіщення (email/SMS) — закладені в архітектурі (розділ 5) як точки розширення, але не обов'язкові зараз.
- Підтримка сотень одночасних користувачів у продакшн-масштабі.

## 3. Контекст: офіційна альтернатива УЗ

Офіційний застосунок «Укрзалізниця: квитки на потяг» має вбудовану функцію моніторингу з опціональним автовикупом (вхід через **Дія.Підпис**): можна відстежувати конкретний поїзд або весь напрямок, обирати тип вагона й розташування місця, і при автовикупі кошти списуються заздалегідь та повертаються автоматично, якщо квиток не знайдено або моніторинг скасовано. Є винятки — авто-викуп не працює на напрямках до/з Польщі (крім «Інтерсіті+») і на прямому потязі Київ–Відень: там доступні лише сповіщення про відкриття продажу.

**Чому все одно варто робити власного бота:**
- Гнучкіша логіка сповіщень (розділ 9) — офіційний застосунок надсилає одне push-сповіщення, а не серію ескалаційних нагадувань.
- Зручність: усе в Telegram, без потреби тримати застосунок на видноті.
- Працює і там, де офіційний автовикуп недоступний (Польща/Відень) — принаймні як сповіщувач.

Офіційна функція технічно швидша (працює на сервері УЗ, без мережевої затримки полінгу ззовні), тож для дуже гарячих напрямків має сенс тримати увімкненими обидва інструменти паралельно.

## 4. Технологічний стек

Архітектура нижче технологічно-нейтральна за задумом, але для конкретики наведено приклад на Node.js — це найпоширеніший вибір для Telegram-ботів із найкращою бібліотечною підтримкою. **Якщо агент впевненіше працює з іншим стеком (Python, .NET, Go) — перенось один-в-один, зберігаючи модулі та їх зони відповідальності з розділу 5.**

| Компонент | Рекомендація | Альтернатива |
|---|---|---|
| Runtime | Node.js 20+, TypeScript (strict mode) | Python 3.11+ / .NET 8 |
| Telegram Bot API | `telegraf` | `node-telegram-bot-api`, `grammy` |
| HTTP-клієнт до УЗ | `axios` / `got` з cookie jar | `httpx` (Python) |
| Fallback-браузер (розділ 7.6) | `playwright` | `puppeteer` |
| База даних | SQLite (`better-sqlite3`) | PostgreSQL — якщо плануєш кількох користувачів |
| Планувальник | внутрішній цикл на `setInterval`/таймерах + `node-cron` для нічного режиму | APScheduler (Python) |
| Логування | `pino` | `winston` |
| Процес-менеджер | Docker + `restart: unless-stopped`, або PM2 | systemd service |

```
│   Telegram API    │◄────────►│   BotHandler (FSM)      │
└──────────────────┘          └────────────┬────────────┘
                                            │
                                            ▼
                              ┌────────────────────────┐
                              │   Repository / SQLite    │
                              │ users · monitors ·        │
                              │ notification_log ·        │
                              │ station_cache             │
                              └────────────┬────────────┘
                                            ▲
                     ┌──────────────────────┼───────────────────────┐
                     │                                              │
         ┌───────────▼─────────────┐                  ┌─────────────▼─────────────┐
         │  MonitoringWorker         │   found event    │       AlertEngine           │
         │  (полінг, N сек/монітор)  │─────────────────►│  (ескалаційні сповіщення,   │
         └───────────┬─────────────┘                  │   окремий модуль!)          │
                     │                                  └─────────────┬─────────────┘
                     ▼                                                ▼
         ┌─────────────────────────┐                    ┌─────────────────────────┐
         │      UzApiClient          │                    │   Telegram sendMessage    │
         │  (HTTP + анти-бот заходи) │                    │   (нове повідомлення,      │
         └───────────┬─────────────┘                    │    не editMessageText)     │
                     │                                    └─────────────────────────┘
                     ▼
         ┌─────────────────────────┐
         │   UzSessionManager         │  ← контролює cookies/токен/капчу (розділ 7.3, 7.6)
         └─────────────────────────┘
```

### Модулі та зони відповідальності

| Модуль | Відповідальність |
|---|---|
| `BotHandler` (FSM) | Діалог з користувачем, команди, inline-кнопки. НЕ звертається до УЗ API напряму — лише через `UzApiClient` (для автодоповнення станцій) і `Repository`. |
| `Repository` | Єдина точка доступу до БД. Ніхто інший не робить SQL напряму. |
| `MonitoringWorker` | Фоновий цикл: читає активні монітори, викликає `UzApiClient`, порівнює результат зі знімком (snapshot), передає подію в `AlertEngine`. |
| `AlertEngine` | **Окремий модуль** (не частина Worker!) — логіка ескалації сповіщень, таймери, умови зупинки. Легко замінити інтервали чи додати канали сповіщень, не чіпаючи Worker. |
| `UzApiClient` | HTTP-запити до booking.uz.gov.ua: пошук станцій, пошук поїздів, вагонів/місць. Інкапсулює заголовки, retry, парсинг відповіді. |
| `UzSessionManager` | Підтримка "сесії" перед УЗ: cookies, токен автентифікації (якщо потрібен — розділ 7.3), ротація User-Agent, реакція на капчу/блокування. |
| `Logger` | Структуровані логи (розділ 13). |
| `ConfigService` | Типізоване читання конфігурації з `.env` / `config.json` — жодних магічних чисел у коді (розділ 9, 10). |

## 6. Модель даних

```sql
CREATE TABLE users (
  id                INTEGER PRIMARY KEY,
  telegram_chat_id  TEXT UNIQUE NOT NULL,
  username          TEXT,
  is_whitelisted    BOOLEAN DEFAULT 1,   -- див. розділ 14 (безпека)
  created_at        DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE monitors (
  id                  INTEGER PRIMARY KEY,
  user_id             INTEGER REFERENCES users(id),
  from_station_id     TEXT NOT NULL,
  from_station_name   TEXT NOT NULL,
  to_station_id       TEXT NOT NULL,
  to_station_name     TEXT NOT NULL,
  travel_date         DATE NOT NULL,
  train_number        TEXT,              -- NULL = "будь-який поїзд на напрямок"
  wagon_types         TEXT,              -- JSON-масив: ["купе","плацкарт"]
  seat_position       TEXT DEFAULT 'any',-- 'lower' | 'upper' | 'any'
  status              TEXT NOT NULL DEFAULT 'active',
                       -- active | paused | found | muted | completed | expired | error
  alert_profile       TEXT DEFAULT 'aggressive', -- 'aggressive' | 'standard' (розділ 9.5)
  found_at            DATETIME,          -- момент першого виявлення поточної "хвилі" квитка
  last_alert_at       DATETIME,
  next_alert_at       DATETIME,
  alert_attempt_count INTEGER DEFAULT 0,
  last_checked_at     DATETIME,
  last_snapshot       TEXT,              -- JSON знімок останньої перевірки
  consecutive_failures INTEGER DEFAULT 0,
  created_at          DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at          DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE notification_log (
  id            INTEGER PRIMARY KEY,
  monitor_id    INTEGER REFERENCES monitors(id),
  sent_at       DATETIME DEFAULT CURRENT_TIMESTAMP,
  stage         TEXT,     -- instant | escalation1 | escalation2 | steady | update | resolved
  message_text  TEXT,
  snapshot      TEXT      -- JSON
);

CREATE TABLE station_cache (
  station_id       TEXT PRIMARY KEY,
  name             TEXT NOT NULL,
  name_normalized  TEXT,
  updated_at       DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

`station_cache` зменшує кількість звернень до автодоповнення УЗ (розділ 7.2) для повторюваних назв станцій.

## 7. Інтеграція з API Укрзалізниці

### 7.1 Головне застереження

Це **неофіційне, недокументоване API**. Усе нижче в цьому розділі — узагальнення з публічних відкритих проєктів (кілька існуючих GitHub-репозиторіїв реалізовували подібних ботів для цього ж сайту), а не офіційна документація. Структура **може відрізнятися на момент реалізації**. Перш ніж писати `UzApiClient`, агент має:
1. Відкрити `booking.uz.gov.ua` у браузері з DevTools → вкладка Network.
2. Вручну виконати пошук станції та пошук поїзда.
3. Зафіксувати реальні URL, метод, заголовки, тіло запиту й форму відповіді.
4. Задокументувати відхилення від цього розділу прямо в коді коментарями.

Пряме автоматичне звернення до сайту з боку сторонніх інструментів (у т.ч. з боку AI-агента під час розробки) з великою ймовірністю буде заблоковане бот-захистом без "браузероподібної" сесії — це очікувано, а не помилка.

### 7.2 Історично відомі ендпоінти (потребують перевірки)

| Призначення | Метод і шлях (орієнтовно) | Примітка |
|---|---|---|
| Автодоповнення станції | `GET /uk/train_search/station/?term={query}` | Повертає масив станцій-кандидатів з їх внутрішнім `station_id`. |
| Пошук поїздів на маршруті/даті | `POST /uk/train_search/list` (точний шлях перевірити) | Тіло: станція відправлення/прибуття (ID), дата. |
| Список вагонів/місць для поїзда | окремий запит за `trip_id` поїзда | Повертає масив вагонів з кількістю вільних місць. |

### 7.3 Автентифікація (умовно обов'язкова — перевірити емпірично)

Деякі новіші неофіційні клієнти показують, що повноцінна робота з API (принаймні для дій ближче до бронювання) вимагає **входу за номером телефону + одноразовим кодом із SMS**, після чого видається токен/сесія з обмеженим часом дії. Невідомо напевно, чи потрібен цей вхід і для *самого лише* пошуку наявності місць, чи тільки для бронювання/оплати.

**Що робити:** реалізувати `UzSessionManager` умовно:
- Спочатку спробувати анонімний пошук (без входу). Якщо працює стабільно — авторизація не потрібна для MVP.
- Якщо анонімний пошук блокується або не повертає повних даних про місця — додати одноразовий діалог у боті: користувач вводить номер телефону, бот пересилає код підтвердження з SMS, `UzSessionManager` зберігає токен/cookies в БД (зашифровано, розділ 14) і оновлює їх до закінчення строку дії.

### 7.4 Орієнтовна форма відповіді (вагони/місця)

Для планування парсера — приклад полів, які історично зустрічалися у відповіді про вагони (**перевірити на актуальних даних, не покладатися сліпо**):

```json
{
  "number": "6",
  "mockup_name": "Вагон типу Купе на 36 місць",
  "seats": [1, 2, 5, 6, 9, 10],
  "free_seats_top": 3,
  "free_seats_lower": 5,
  "price": 164938,
  "air_conditioner": true
}
```

Важливі нюанси:
- `price`, ймовірно, у копійках — ділити на 100 для гривень (перевірити).
- `free_seats_top` / `free_seats_lower` дають основу для фільтра "тільки нижні місця" (розділ 11).
- `mockup_name` — готовий людський опис вагона, зручно вставляти прямо в текст сповіщення.

### 7.5 HTTP-заголовки

Обов'язково виставляти реалістичні заголовки браузера, інакше типовий `HttpClient`/`curl` отримає 403:
- `User-Agent` — актуальний рядок Chrome/Firefox.
- `Accept: application/json`, `Accept-Language: uk-UA,uk;q=0.9`.
- `Referer: https://booking.uz.gov.ua/`.
- `Sec-Ch-Ua`, `Sec-Fetch-Mode`, `Sec-Fetch-Site` — клієнтські хінти Chrome.
- Cookie jar: спершу завантажити головну сторінку, зберегти видані cookies, використовувати їх у наступних запитах у межах однієї "сесії".

### 7.6 Анти-бот захист: реалістична стратегія

Підтверджено (як власним тестовим зверненням при підготовці цього документа, так і згадками в публічних проєктах), що сайт має Cloudflare-подібний захист **і** reCAPTCHA, яка може з'являтися періодично навіть у легітимних сесіях. Це означає: **чистий HTTP-полінг рано чи пізно впреться в капчу**, скільки б заголовків ви не підробили. Реалістичний план:

1. **Основний шлях:** `UzApiClient` через звичайний HTTP-клієнт з коректними заголовками й cookie-сесією — працює, поки капча не з'явилась.
2. **Fallback:** якщо кілька перевірок поспіль повертають HTML-сторінку виклику замість JSON (ознака капчі/блоку) — переключитись на `Playwright` з persistent-профілем браузера, у якому користувач один раз вручну проходить капчу.
3. **Сповіщення про капчу:** якщо навіть Playwright-сесія впирається в капчу, яку неможливо пройти автоматично — надіслати користувачу окреме службове повідомлення ("⚠️ УЗ показує капчу, монітор X призупинено, розв'яжи вручну і напиши /resume_X") замість того, щоб мовчки переставати працювати.
4. Не намагатися програмно **розв'язувати** капчу (обхід CAPTCHA-сервісів тощо) — це виходить за межі цього проєкту.

### 7.7 Rate limiting та захист від бану IP

- Глобальний ліміт запитів **по всіх моніторингах разом** (не на кожен монітор окремо) — напр. не частіше 1 запиту/секунду сумарно, через чергу.
- Базовий інтервал перевірки одного монітора: 60–120 секунд (конфігурований).
- Нічне уповільнення (01:00–05:00, конфігурується) — **тільки для стадії пошуку** (коли квиток ще не знайдено). Як тільки статус монітора `found`, нічне уповільнення **не застосовується** — Alert Engine працює на повній швидкості цілодобово (розділ 9).
- Експоненційний backoff при мережевих помилках: старт 5с, максимум 5 хв, після N (напр. 10) невдач поспіль — статус `error` + одноразове повідомлення користувачу.

## 9. ⚠️ Alert Engine — критична система сповіщень

**Це ядро продукту й головна відмінність від "звичайного" бота-нагадувача. Реалізується як окремий модуль, незалежний від `MonitoringWorker`.**

### 9.1 Філософія

Типовий підхід — надіслати одне повідомлення й зупинити монітор — **тут навмисно не використовується**. Причина: користувач може спати, не чути телефон, або не помітити перше повідомлення серед інших чатів — а дефіцитний квиток може зникнути за хвилини. Тому головний пріоритет — **будь-якою ціною привернути увагу**, поки квиток ще існує. Багаторазові повідомлення про один і той самий квиток — це **навмисна функція, а не помилка чи спам**.

### 9.2 Явно ЗАБОРОНЕНО (типові "спрощення", яких не повинно бути)

- ❌ Надіслати одне повідомлення і зупинити/поставити на паузу монітор.
- ❌ Автоматично перевести монітор у статус `completed`/`found` як фінальний одразу після першого сповіщення.
- ❌ Будь-яка дедуплікація чи придушення повторних однакових сповіщень.
- ❌ **Редагувати** попереднє повідомлення (`editMessageText`) замість надсилання нового (`sendMessage`) — редагування не гарантує повторний звук/push на пристрої користувача, а саме це і є метою.
- ❌ Нічний "тихий режим" для вже знайденого квитка за замовчуванням (може існувати як опція, але вимкнена за замовчуванням — розділ 9.5).
- ❌ `disable_notification: true` для будь-якого повідомлення Alert Engine.

### 9.3 Логіка ескалації

| Стадія | Часове вікно від першого виявлення | Частота повідомлень | Позначка |
|---|---|---|---|
| Instant | t = 0 | 1 повідомлення негайно | 🚨 |
| Escalation 1 | 0–5 хв | кожні 20–30 сек (за замовчуванням 25 сек) | 🔥 |
| Escalation 2 | 5–30 хв | кожну 1 хв | 🔔 |
| Steady | 30 хв+ | кожні 5 хв, безкінечно | 🔁 |

Плюс: **позапланове** повідомлення одразу, якщо снапшот змінився суттєво (кількість місць зменшилась/збільшилась, з'явився/зник тип вагона) — незалежно від того, чи настав час за розкладом.

Усі числа — з конфігурації (`.env`), не хардкод:

```env
ALERT_INSTANT_DELAY_SEC=0
ALERT_ESCALATION1_INTERVAL_SEC=25
ALERT_ESCALATION1_DURATION_SEC=300
ALERT_ESCALATION2_INTERVAL_SEC=60
ALERT_ESCALATION2_DURATION_SEC=1500
ALERT_STEADY_INTERVAL_SEC=300
ALERT_NIGHT_QUIET_ENABLED=false
```

### 9.4 Псевдокод

```
function onCheckResult(monitor, result):
    wasFound = monitor.status == 'found'
    isFound  = result.matchingPlaces > 0   # з урахуванням фільтрів монітора

    if not wasFound and isFound:
        # НОВА подія виявлення — навіть якщо це повторна поява після зникнення
        monitor.status = 'found'
        monitor.found_at = now()
        monitor.alert_attempt_count = 0
        monitor.last_snapshot = result.snapshot
        sendNewMessage(monitor, result, stage='instant')
        monitor.alert_attempt_count += 1
        monitor.next_alert_at = now() + ESCALATION1_INTERVAL

    elif wasFound and isFound:
        if snapshotChanged(monitor.last_snapshot, result.snapshot):
            sendNewMessage(monitor, result, stage='update')
            monitor.last_snapshot = result.snapshot
        if now() >= monitor.next_alert_at:
            stage = currentStage(monitor.found_at)   # escalation1 | escalation2 | steady
            sendNewMessage(monitor, result, stage=stage)
            monitor.alert_attempt_count += 1
            monitor.next_alert_at = now() + intervalFor(stage)

    elif wasFound and not isFound:
        # Дебаунс: підтвердити зникнення ДВОМА поспіль негативними перевірками,
        # щоб не скидати ескалацію через одноразовий "глюк" відповіді API
        monitor.negative_checks += 1
        if monitor.negative_checks >= 2:
            monitor.status = 'active'   # повертаємось у звичайний пошук, НЕ completed
            monitor.found_at = null
            sendNewMessage(monitor, message='Квиток зник, продовжую моніторинг', stage='resolved')

    else:
        pass  # звичайна перевірка, нічого не знайдено — без сповіщень
```

**Умови зупинки повторних сповіщень** (лише вони, більше жодних):
1. Квиток більше не відповідає фільтрам (після дебаунсу вище) — монітор повертається в активний пошук.
2. Користувач натиснув «🔕 Досить нагадувань» → `status = 'muted'` (моніторинг триває мовчки; якщо зникне і з'явиться знову — нова хвиля сповіщень стартує з Instant).
3. Користувач видалив монітор.
4. Користувач натиснув «✅ Я купив(ла)» → `status = 'completed'`.

### 9.4.1 Шаблони повідомлень (приклади для message-компонування)

**Instant:**
```
🚨 Знайдено квиток!
Поїзд №091К Київ — Львів, 25.07.2026
```

**Escalation / Steady (повторне, з відміткою змін):**

### 9.5 Alert Profiles (закладено на майбутнє, не для зміни поточної поведінки)

`AlertEngine` читає профіль з `monitor.alert_profile`. Зараз є сенс реалізувати лише `aggressive` (все, що описано вище) — саме він за замовчуванням для нових моніторингів. Структуру варто закласти так, щоб пізніше можна було додати `standard` (рідші нагадування) чи `night-safe` без переписування ядра.

## 10. Background Worker і полінг

- Один цикл на процес, що періодично (напр. кожні 5–10 сек) перевіряє, чи для якогось монітора настав час чергової перевірки (`last_checked_at + interval <= now`), і чи для якогось монітора настав час Alert Engine (`next_alert_at <= now`) — це **дві незалежні шкали часу**, що не заважають одна одній.
- Черга запитів до `UzApiClient` з єдиним rate-лімітером на все (розділ 7.7), навіть якщо активних моніторингів кілька.
- При старті процесу: підвантажити з БД усі монітори зі статусом `active`, `found`, `muted` і відновити для них цикл (для `found`/`muted` — не скидати `found_at`, продовжити ескалацію з поточної стадії, а не з Instant).
- Retry/backoff — розділ 7.7.

## 11. Фільтри та кастомізація

- **Типи вагонів:** Купе, Плацкарт, Люкс/СВ, Сидячий, Інтерсіті — мультивибір при створенні монітора.
- **Розташування місця:** нижнє / верхнє / будь-яке (якщо API повертає `free_seats_top`/`free_seats_lower` окремо — розділ 7.4).
- **Кілька моніторингів одночасно** для одного користувача (кожен — окремий маршрут+дата+поїзд+фільтри). М'який ліміт — напр. 20 активних, щоб не перевантажити власний rate-limit.
- **Кілька дат:** реалізується як кілька окремих моніторингів (по одному на дату) в MVP; підтримка "діапазону дат" одним монітором — Phase 2 (розділ 19).

## 12. Обробка помилок і Edge Cases

| Ситуація | Очікувана поведінка |
|---|---|
| УЗ повертає 403 / HTML замість JSON (капча/блок) | `consecutive_failures++`, backoff; після N поспіль → `status='error'`, одне повідомлення користувачу, продовжувати ретраї з більшим інтервалом. |
| Структура відповіді УЗ змінилась (несподівані/відсутні поля) | Парсер не падає весь процес — ловить помилку, логує сирий JSON, трактує цикл як "даних немає", інкрементує лічильник збоїв. |
| Telegram `sendMessage` → 403 (бот заблокований користувачем) | Позначити користувача неактивним, зупинити воркер для його моніторингів. |
| Telegram `sendMessage` → 429 | Пошанувати `retry_after`, повторити відправку пізніше (не втратити повідомлення). |
| Дата поза типовим вікном продажу квитків | Попередити при створенні монітора, але дозволити зберегти — продаж може відкритися пізніше. |
| Немає такого поїзда на вказану дату/маршрут | Повідомити одразу в FSM, запропонувати "будь-який поїзд" замість конкретного номера. |
| Дублікат монітора (однакові from/to/date/train) | Попередити й запитати: оновити існуючий чи створити ще один. |
| Рестарт сервісу/контейнера | При старті відновити всі активні монітори з БД (див. розділ 10). |
| Ticket flapping (зникає й одразу з'являється) | Дебаунс два негативні цикли поспіль перед переходом з `found` в `active` (розділ 9.4). |
| Часовий пояс | Усюди `Europe/Kyiv`. |

## 13. Логування

- Структуровані JSON-логи, рівні `debug/info/warn/error`.
- Кожна спроба полінгу: `monitor_id`, стислий результат, час відповіді.
- Кожне надіслане сповіщення — і так пишеться в `notification_log` (розділ 6), додатково дублювати в лог-файл на рівні `info`.
- Ротація/обмеження розміру лог-файлів.
- Критичні збої (тривале блокування, помилки БД) — окремим Telegram-повідомленням адміну/користувачу, а не лише в лог-файл, який ніхто не читає в реальному часі.

## 14. Безпека

- Токен Telegram-бота — тільки в змінних середовища/секретах, ніколи в репозиторії.
- **Білий список `chat_id`** — бот ігнорує повідомлення від будь-кого, крім дозволених `telegram_chat_id` з таблиці `users` (це особистий інструмент, не публічний бот).
- Токени/cookies сесії УЗ (якщо є, розділ 7.3) зберігати в БД у зашифрованому вигляді, не у відкритому тексті.
- Не логувати токени й номери телефонів у відкритому вигляді.
- Long polling для Telegram (не webhook) — не потрібен публічний HTTPS-ендпоінт, простіше й безпечніше для персонального проєкту.

## 15. Продуктивність і масштаб

Орієнтир — **до ~20–50 одночасних моніторингів на одному процесі**, не enterprise-масштаб. Це свідомо: єдиний Node.js-процес і один файл SQLite повністю покривають потребу особистого бота; ускладнення на кшталt мікросервісів чи брокерів повідомлень тут — over-engineering, а не перевага.

- Затримка від виявлення квитка до відправки Instant-повідомлення: ціль < 2 сек.
- Реакція бота на команди користувача: < 1 сек (не рахуючи мережевих запитів до УЗ).

---

## 16. Тестування

- **Unit:** парсер відповіді `UzApiClient` (у т.ч. на пошкоджений/змінений JSON); переходи станів `AlertEngine` з фейковими таймерами (перевірити саме розклад із розділу 9.3); переходи FSM бота.
- **Інтеграційний сценарій:** мок УЗ API, що послідовно повертає "немає квитків" → "з'явились" → "кількість змінилась" → "зникли" — перевірити точну послідовність і кількість надісланих повідомлень (включно з тим, що жодне не було редагуванням).
- **Ручний чекліст:** повний цикл створення монітора через реального бота, пауза/відновлення, видалення, натискання "Досить нагадувань".

- Жодних магічних чисел для інтервалів/порогів — усе через `ConfigService`.
- `README.md` з інструкцією: як отримати токен бота, як запустити локально, як задеплоїти.

## 18. Розгортання

- **Обов'язково always-on машина** (VPS, Raspberry Pi, домашній сервер) — ноутбук, що засинає, зупиняє моніторинг саме тоді, коли він найпотрібніший.
- Docker + `docker-compose` (бот + volume для файлу SQLite).
- Автоперезапуск при падінні (`restart: unless-stopped` або systemd).
- Приклад `.env`:

```env
TELEGRAM_BOT_TOKEN=xxxxx
ALLOWED_CHAT_IDS=123456789
DATABASE_PATH=./data/bot.db
UZ_BASE_URL=https://booking.uz.gov.ua
POLL_INTERVAL_SEC=90
NIGHT_THROTTLE_ENABLED=true
NIGHT_THROTTLE_START=01:00
NIGHT_THROTTLE_END=05:00
NIGHT_THROTTLE_INTERVAL_SEC=600
LOG_LEVEL=info
```

(плюс усі `ALERT_*` змінні з розділу 9.3)

## 19. Поетапний план реалізації

1. **Milestone 1 — Розвідка API.** Вручну через DevTools зафіксувати реальні ендпоінти/заголовки/форму відповіді (розділ 7). Без цього кроку решта неможлива.
2. **Milestone 2 — Кістяк.** БД + `Repository`, `UzApiClient` з базовим пошуком станцій/поїздів, без Telegram.
3. **Milestone 3 — Бот і FSM.** Повний діалог створення монітора (розділ 8), збереження в БД.
4. **Milestone 4 — Worker + просте сповіщення.** Полінг працює, при знайденому квитку — поки що одне повідомлення (тимчасово, для перевірки самого полінгу).
5. **Milestone 5 — Alert Engine.** Заміна "одного повідомлення" на повну ескалаційну логіку розділу 9. Це ключовий milestone — без нього проєкт не відповідає вимогам.
6. **Milestone 6 — Стійкість.** Retry/backoff, обробка капчі/блокувань (7.6), edge cases (розділ 12).
7. **Milestone 7 — Полірування.** Логи, тести, README, Docker.

## 21. Відповідальне використання

Це інструмент для власного планування поїздок, а не для перепродажу квитків. Дотримуйтесь помірної частоти запитів (розділ 7.7), щоб не створювати зайве навантаження на систему УЗ і не отримати блокування. Це неофіційна інтеграція зі стороннім сайтом — умови використання booking.uz.gov.ua можуть обмежувати автоматизовані звернення; використання — на власний розсуд. Там, де підходить офіційний вбудований моніторинг з автовикупом (розділ 3), він швидший за конструкцію "ззовні" й вартий паралельного використання.
